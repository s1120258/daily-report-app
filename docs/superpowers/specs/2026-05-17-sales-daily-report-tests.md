# 営業日報システム テスト仕様書

作成日: 2026-05-17

---

## 1. テスト方針

### 目的

- API の権限制御（ロール別アクセス）が正しく動作することを保証する
- 日報・訪問記録のCRUDが仕様どおりに動くことを保証する
- 主要なユーザー操作フローが壊れていないことをE2Eで保証する

### テスト種別と範囲

| 種別 | 対象 | 目的 |
| --- | --- | --- |
| 統合テスト（API） | Route Handlers | 権限制御・ビジネスロジック・DBとの連携 |
| E2Eテスト | ブラウザ操作 | 主要ユーザーフローの動作確認 |

> 単体テストはユーティリティ関数・バリデーション関数が増えた際に追加する。現時点では統合テスト・E2Eテストを優先する。

### 使用ツール

| ツール | 用途 |
| --- | --- |
| Jest | 統合テスト（API Route Handlers） |
| Supertest | HTTPリクエストのシミュレーション |
| Playwright | E2Eテスト |
| Prisma（テスト用DB） | 統合テスト時にテスト用PostgreSQLへ接続 |

## 2. テストデータ方針

### 共通フィクスチャ

各テストスイート実行前に以下のデータをシードする。

| ID | ロール | 名前 | メール |
| --- | --- | --- | --- |
| 1 | manager | テスト部長 | manager@test.com |
| 2 | sales | テスト営業A | sales-a@test.com |
| 3 | sales | テスト営業B | sales-b@test.com |

| ID | 顧客名 |
| --- | --- |
| 10 | 株式会社テストA |
| 11 | 株式会社テストB |

### テスト用DB

- 統合テストは専用のテスト用PostgreSQLに接続する（`.env.test` で管理）
- 各テストケース前後にテーブルをクリーンアップし、テスト間の干渉を防ぐ
